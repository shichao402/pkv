# CLI 发布执行 Skill

适用于用户直接说“发布”“发版”“release”这类指令的场景。
默认目标不是讨论流程，而是把本地这半段发布动作真正执行完。

## 用户意图默认解释

当用户只说 `发布`，且没有额外限定时，默认按下面的语义理解：

- 这不是单纯要一个方案说明，而是要 agent 实际执行发布前半程。
- 如果项目采用 version-driven release，agent 应尽量自主完成本地动作。
- 默认发布级别为 PATCH bump。
- 如果当前仓库状态显示版本已经提升或 release commit 已经存在，agent 应沿用现状继续推进，而不是机械地重新 bump。

## 默认执行目标

agent 应根据仓库状态，自动落到以下三种结果之一：

1. `提升版本号 -> 提交 -> 推送`
2. `版本号已改好 -> 提交 -> 推送`
3. `release commit 已存在但未推送 -> 直接推送`

除非遇到真正阻塞的风险点，否则不要停在“我建议你接下来执行这些命令”。

## 默认检查项

在执行前，至少检查：

- 单一版本来源，例如 `version.json`、`package.json`、`pyproject.toml`、`Cargo.toml`
- 发布 workflow、CI 配置或 release 脚本，确认发布是如何触发的
- 当前分支、默认远端、tracking 状态
- `git status --short`
- 本地与远端 tag 是否已占用目标版本
- 当前分支是否已经 ahead，且是否已经包含 release commit

## 决策顺序

```text
用户说“发布”
  -> 识别版本源与 CI 触发方式
  -> 检查 git status / branch / remote / ahead-behind / tags
  -> 已有未推送的 release commit？
      -> 是：直接 push
      -> 否：继续
  -> 版本文件已经被改成合法的新版本但尚未提交？
      -> 是：沿用该版本并提交
      -> 否：默认 bump patch
  -> 只 stage 本次发布应该纳入的文件
  -> commit
  -> push
  -> 汇报版本、commit、分支、CI 接手方式
```

## 默认版本策略

- 用户只说 `发布`：默认执行 PATCH bump。
- 用户明确说 `发布 minor`、`发布 major`、`发 beta`：按用户要求执行。
- 如果版本文件已经是一个合法且更高的新版本：沿用该版本，不重复 bump。
- 如果 HEAD 已经包含匹配目标版本的 release commit，且只是尚未推送：直接 push，不创建第二个 release commit。
- 如果目标版本 tag 在本地或远端已经存在：继续寻找下一个可用版本；如果无法安全决策，再说明冲突。
- 不要因为没有 CHANGELOG 就默认停止，除非仓库明确把它作为发版前置条件。

## Agent 执行准则

### 1. 先检查，再行动

不要在未查看仓库状态时直接修改版本号。
先确定版本源、发布触发条件、当前分支状态，再决定是 bump、commit，还是直接 push。

### 2. 默认完成本地发布半程

如果仓库采用 version-driven release，agent 的默认职责包括：

- 提升版本号
- 提交
- 推送

下面这些动作默认不应留给用户手工补做：

- 手动判断下一个 PATCH 版本
- 手动执行 `git add`
- 手动编写 release commit
- 手动执行 `git push`

### 3. 发现已有进度时要复用，而不是重做

如果仓库已经处于下面任一状态，应直接承接现有进度：

- 版本号已经修改但尚未提交
- 已经创建了 release commit，但尚未推送
- 当前分支 ahead 远端，且 release 相关提交已经存在

不要在这种情况下再次 bump 版本，制造重复发布提交。

### 4. 谨慎处理工作区内容

- 永远不要回滚用户已有修改。
- 默认必须提交权威版本文件。
- 如果工作区里已有明显属于本次发布的源代码改动，应识别并一并纳入，而不是只提交版本号。
- 对明显的派生文件、构建产物、IDE 同步目录、缓存目录，不要因为它们出现在工作区里就自动纳入提交。
- 如果工作区混有高风险且互不相关的改动，无法安全判断是否应该一起发布，先指出冲突再询问用户。

### 5. 不要被辅助脚本的边界限制住

如果仓库里存在 release helper script，而它只负责：

- 修改版本号
- 跑测试
- 构建产物

但没有负责 commit 和 push，那么 agent 仍然要继续完成 commit 和 push。
不要因为脚本结束了，就把最后一步留给用户。

### 6. 只有在真正阻塞时才停下来问

仅在下面情况才需要先问用户：

- 无法判断应做 `major`、`minor` 还是 `patch`
- 工作区混有明显互相无关、且影响推送安全性的改动
- 仓库没有可识别的版本源
- 推送目标分支不明确
- 推送被权限、网络或保护分支策略阻塞

除此之外，应尽量自行完成。

## 提交与推送约定

推荐默认值：

- commit message: `chore: release vX.Y.Z`
- push branch: 当前会触发发布的目标分支；如果仓库约定固定为 `main`，优先推送 `main`
- 如果本地已经在 `main` 且远端为 `origin`，直接 push，不要反复询问
- 如果本地 tracking 已经表明分支 ahead，优先判断是否只差一次 push

## 与 Rule 的分工

- `cli-release-rules`：只负责版本源、构建注入、单一版本来源约束
- `cli-release-workflow`：负责 agent 在收到“发布”指令后应执行哪些实际动作

如果问题是“版本号该放哪里”，优先看 rule。
如果问题是“现在帮我发布”，优先按这个 skill 直接执行。

## 输出要求

完成后至少反馈：

- 使用了哪个版本源
- 从哪个版本提升到哪个版本，或说明沿用了哪个已存在的新版本
- 创建了哪个 commit，或说明复用了哪个已有 release commit
- 是否已经 push
- 哪个 CI 或 workflow 会接手后续发布
- 如果没能 push，给出精确阻塞原因和下一条用户可执行命令

## 反例

以下行为不符合本 skill 的预期：

- 只告诉用户“你可以改 version.json 再提交”
- 只提升版本号，不提交
- 只提交，不尝试推送
- 已经有 release commit 还再造一个新的版本提交
- 遇到普通 patch 发布也先问一轮“你想升哪个版本”
- 仓库明明是 CI 自动发版，却仍要求用户手动打 tag、手动上传产物
