# Vikunja 项目推进工作流

## 项目配置

- **项目**：`PKV`（所有条目统一在此，用 `bug`/`improvement`/`feature` 标签区分类型）
- **任务文件目录**：`docs/task`（每个条目的执行计划，命名格式 `VK-<id>.md`）

`PKV` 是默认目标，不是绝对真值。凡是查询不到项目、条目或目标时，先排查命名偏差、归档状态、父子项目层级，再决定是否真的“不存在”。

## 推进生命周期

```
1. 选条目  ── Vikunja Backlog 按优先级选取
       ↓
2. 建 Plan ── 在 docs/task/VK-<id>.md 创建执行计划
       ↓       Vikunja 状态 → 执行中
3. 实现    ── 按 plan 编码
       ↓
4. 验证    ── 达到停止条件（能跑、能验、不卡下一步）
       ↓
5. Review  ── 审查代码变更，确认无遗漏
       ↓
6. 回写    ── 更新架构文档 + task plan
       ↓
7. 提交    ── commit [VK-<id>] + push
       ↓
8. 闭环    ── Vikunja 条目 → 已完成，附 commit hash
```

## 选条目规则

通过 Vikunja MCP 工具查询 `PKV`，按优先级排序，选择第一个满足以下条件的条目：

- 标签含 `执行中` 或 `可开工`
- **不含** `观察中` 标签
- 不是纯细化/优化

**单条目原则**：每轮只选取并推进**一个**条目。仅在以下条件全部满足时允许并行：
- 各条目互不依赖，改动模块完全不重叠
- 每个条目都是简单修复（one-liner、配置调整），不涉及架构变更
- 在 plan 阶段明确列出并论证了可并行的理由

## 查询失败与歧义处理

当 `PKV`、目标条目或搜索结果“不存在”时，不要直接下结论，按下面顺序处理：

1. 先检查是否是命名问题：大小写、缩写、前后缀、空格、连字符、父子项目路径是否有偏差
2. 再检查是否是可见性问题：项目是否已归档、是否在子项目下、是否当前筛选条件过严
3. 如果能找到高度相似的候选项，向用户明确确认，而不是自行假设
4. 只有在完成上述核对后仍无结果，才说明“当前未找到对应目标”

推荐提问方式：

- “我没有查到 `PKV` 这个项目。是否可能是项目名写错，或者它已经归档/移动到子项目下？”
- “我没有查到这个目标。你说的是不是另一个命名，或者需要我先按关键词搜索相近条目？”

禁止行为：

- 因为精确名称未命中，就直接判定 backlog 为空
- 未经确认就擅自切换到名称相似的其他项目
- 把“没查到”表述成“确定不存在”

## 建 Plan

在 `docs/task/VK-<id>.md` 创建执行计划，内容包括：

- 条目标题和 Vikunja 链接
- 本轮停止条件
- 实施步骤
- 测试口径
- 涉及的文件和模块

创建 plan 后，将 Vikunja 条目标签更新为 `执行中`。

## 闭环动作

完成后执行以下两步（缺一不可）：

1. `git commit` 格式：`<type>(<scope>): <description> [VK-<id>]`
2. 在 Vikunja 将条目标记为 `done`，描述中补充 commit hash 和改动摘要

## Issue-First 原则

发现 bug、改进点时：

1. **先在 `PKV` 创建任务，打上 `bug` 或 `improvement` 标签**，不直接修复
2. 创建前先搜索避免重复——同现象同根因则追加到已有 issue
3. 只有当前推进条目内的问题、或 P0 阻塞时，才在同一轮直接处理

## 状态标签约定

| 标签 | 含义 |
|------|------|
| `执行中` | 当前主线，优先推进 |
| `可开工` | 无更高优先级阻塞时可接手 |
| `先规划` | 先出方案，本轮不深挖实现 |
| `观察中` | 只在出现真实问题时才推进 |
| `阻塞中` | 依赖外部条件，不主动推进 |
| `已完成` | 当前阶段达标，转维护态 |

优先级：`P0`（紧急）> `P1`（高）> `P2`（普通）

## 推进原则

- 先规划，再动手
- 先框架，再细节
- 先跑通，再优化
- 一轮只收口一个里程碑
- 如果当前里程碑已基本达标且不阻塞后续，切到下一条主线，不继续局部抛光
- 没有 failing sample、没有明确验收缺口时，不默认追加微优化
