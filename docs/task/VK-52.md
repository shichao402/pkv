# VK-52 `bw sync` 在同一 PKV 会话里缺少短时复用，重复命令会触发不必要的网络同步

- Vikunja: `PKV#52`
- Project: `PKV` (ID `8`)
- Status: `已完成`

## 本轮停止条件

- 同一进程、同一有效 `BW_SESSION` 下，短时间内重复执行读取命令时不会每次都调用 `bw sync`
- 发生 `CreateItem` / `EditItem` / `DeleteItem` 等写操作后，后续 `Sync` 不会误跳过
- 自动化测试覆盖同步复用与失效路径
- README 或任务记录说明该行为边界

## 选择原因

- 仓库当前在 Vikunja `PKV` 项目没有现成未完成任务，本轮按 `PROJECT_SUMMARY.md` 的“会话优化”路线补一个可独立交付的增量
- 交互模式已经复用 `BW_SESSION`，但所有资源命令仍会重复触发 `bw sync`，这是同一用户路径里最直接可感知的额外等待
- 该改动可以局限在 `internal/bw` 与少量调用点，风险和验证范围都比较可控

## 实施步骤

1. 梳理 `client.Sync` 与各类写操作路径，确定最小缓存和失效接入点
2. 为 `bw sync` 增加同进程短时复用，只在同一会话且缓存未失效时跳过重复同步
3. 确保 Bitwarden 写操作成功后统一失效缓存，避免后续读取误用旧同步结果
4. 补充针对同步复用、跨会话不复用、写后失效的自动化测试
5. 更新 README / 结果记录，说明行为边界和验证结果

## 测试口径

- `go test ./internal/bw ./internal/key`
- 如命令路径受影响，补跑 `go test ./...`

## 涉及文件和模块

- `internal/key/bitwarden.go`
- `README.md`

## 结果记录

- 为 `internal/bw.Client.Sync` 增加了进程内短时复用窗口，同一 `BW_SESSION` 且最近刚成功同步时会跳过重复的 `bw sync`
- `CreateItem`、`EditItem`、`DeleteItem` 成功后都会统一失效该缓存，避免后续读取继续复用旧同步结果
- SSH key 创建路径改为复用 `bw.Client.CreateItem`，避免该写操作绕过同步缓存失效逻辑
- README 已补充交互模式下“只读短时复用、写后重新同步”的边界说明
